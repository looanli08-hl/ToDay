# ToDay 商业化草案 v0.1

## 1. 产品定位

- 产品名：ToDay（临时名）
- 一句话定位：把生活片段、状态变化和关键记录整理成一条可以回看的私人时间线。
- 当前形态：iPhone 本地优先应用，先不依赖后端。
- 商业化目标：先验证用户是否愿意为“更清晰地理解自己的一天与生活节奏”付费，再逐步扩展到连续洞察和长期报告。
- 产品主线：生活记录与回看。
- HealthKit 定位：辅助数据源，不是产品本身的核心卖点。

## 2. 目标用户

- 核心用户：愿意记录生活、回看状态、理解自己节奏的人。
- 第一批最可能付费的人：
  - 有轻量记录习惯的人
  - 希望回看一天发生了什么、自己状态如何变化的人
  - 想把碎片化记录整合成更清晰生活轨迹的人
- 暂不优先的人：
  - 只想看运动或健康指标图表的人
  - 对记录和回看完全没有需求的人

## 3. 核心价值承诺

- 不是把更多数据堆给用户。
- 而是把今天的生活片段、状态、情绪和关键事件，整理成一条可理解的时间线。
- 用户购买的不是“记录工具”，而是：
  - 更快看懂今天
  - 更容易回看自己的生活轨迹
  - 更容易发现自己的状态模式
  - 更容易持续记录与复盘

## 4. 建议的免费 / 付费分层

### 免费层

- 今日时间线基础查看
- 手动快速记录情绪 / 备注
- 当日基础统计
- 最近 3 天回看
- 基础生活标签或事件记录

### Pro 订阅层

- 7 天 / 30 天 / 月度连续洞察
- 自动生成“今天摘要”
- 状态模式识别：情绪、节奏、活动、记录内容之间的关联提示
- 更完整的历史回看
- 自定义标签、复盘模板
- 后续多设备同步与备份
- 后续网页端联动与账号同步

## 5. 第一阶段可收费功能建议

最先建议卖的不是“更多记录入口”，而是“更高质量的回看和洞察”。

优先级建议：

1. 今日摘要生成
2. 近 7 天生活模式总结
3. 情绪 / 节奏波动提示
4. 更长历史回看与 Pro 解锁

原因：

- 用户更容易为结果付费，而不是为输入动作付费。
- “自动理解今天”比“让我再多填一个表单”更接近真正的付费点。
- 如果以后加入网页端或登录体系，Pro 的价值会进一步增强。

## 6. 建议定价

先用简单订阅验证，不做复杂套餐。

- 月订阅：￥18
- 年订阅：￥128

说明：

- 这个区间适合先做 MVP 验证，不是最终价格。
- 年付需要明显更划算，帮助拉高现金流和留存。
- 如果后续用户明显偏“轻工具”，可以改成更低价位；如果偏“持续洞察”，可以把年价往上抬。
- 后续可以评估是否推出买断制，但不建议在第一阶段同时上线订阅和买断两套复杂策略。

## 7. 关键转化路径

建议的第一版转化路径：

1. 用户首次打开
2. 完成 HealthKit 授权或手动快速记录
3. 当天看到第一条完整时间线
4. 在用户想看“最近 7 天趋势”或“自动总结”时触发付费墙
5. 订阅后立即看到总结结果，而不是只看到一张价格页

付费触发点建议：

- 尝试查看 7 天趋势时
- 想查看自动生成的今日总结时
- 当用户已连续使用 3 天后

## 8. 第一阶段必须补齐的上架资料

- App 对外名称
- 副标题
- 简短卖点文案
- 支持邮箱
- 隐私政策 URL
- 服务条款 URL
- 应用图标
- App Store 截图
- 订阅名称、价格和权益描述
- 官网或落地页 URL

## 9. 法律和合规重点

- 明确说明使用了 HealthKit 数据
- 明确说明数据默认本地优先，不会在未说明情况下上传
- 隐私政策里要写清楚：
  - 收集什么数据
  - 数据存在哪里
  - 是否用于分析
  - 是否与第三方共享
- 不要把产品描述成医疗诊断工具

## 10. 默认品牌文案草案

- 中文名候选：ToDay / 今日线 / 今日感 / 一日线
- 英文副标题候选：Your private timeline for how today actually felt.
- 中文短描述候选：把生活片段、状态和情绪整理成一条真正能看懂的今天。

## 11. 市场与版本策略

- 市场策略：先做中文版本，但产品结构从一开始就预留国际化能力。
- 第一阶段：中文用户、中文文案、人民币定价。
- 第二阶段：补英文文案、国际支付与地区化文案、全球市场版本。
- 开发建议：
  - 文案层从一开始避免写死在代码中
  - 设计上避免只适合中文的极窄布局
  - 数据结构上预留账号、同步和网页端接入能力

## 12. 需要你最终拍板的信息

- 上架主体：个人还是公司
- 品牌名是否继续使用 ToDay 作为临时名
- 是否默认走订阅制
- 支持邮箱用哪个
- 是否准备域名用于官网、支持页和未来网页端
- 是否计划后续接入账号和同步
