# ToDay 产品进化设计：全天自动日记 + 生活回响

**日期**: 2026-03-25
**状态**: 已确认，待实现
**方案**: A（渐进演化）— 在现有 ToDay 代码基础上扩展

---

## 1. 产品定位

一个全天自动记录生活的智能日记，客观还原你的一天，在对的时刻让你重新遇见自己。

### 核心体验

- **时间线** — 客观完整的一天记录（健康、屏幕时间、消费、位置、照片）
- **快门 (Shutter)** — 随时捕捉生活碎片（文字/语音/拍照/视频），极致轻量
- **仪表盘 (Dashboard)** — 多维度当日汇总 + 趋势洞察
- **Echo 回响** — 适时唤醒过去的灵光一现 + 基于生活状态的关怀推送

### 核心原则

- **客观记录为主体，主观权归用户** — AI 辅助生成日记时不加主观判断
- **能自动采集的绝不让用户手动输入，无法自动采集的让手动输入尽可能轻量**
- **快门如相机快门般迅速** — 从点击到开始记录不超过 2 步

### 产品路线

1. **阶段一（当前）**: iOS 端 MVP，验证产品逻辑
2. **阶段二**: 桌面端（管理 + 开放中心），数据同步
3. **阶段三**: MCP 扩展性，更多数据源接入
4. **后续**: 产品重命名，架构重构（Cloud-First）

## 3. 数据模型

### 3.1 新增：ShutterRecord（快门记录）

```
ShutterRecord
├── id: UUID
├── createdAt: Date
├── location: CLLocationCoordinate2D?
├── type: ShutterType  (text / voice / photo / video)
├── textContent: String?
├── mediaURL: URL?
├── voiceTranscript: String?
├── duration: TimeInterval?
└── echoConfig: EchoConfig?
    ├── frequency: EchoFrequency  (high / medium / low / off)
    └── customRemindAt: Date?     (用户自定义提醒时间)
```

### 3.2 新增：ScreenTimeRecord（屏幕时间）

```
ScreenTimeRecord
├── dateKey: String
├── totalScreenTime: TimeInterval
├── appUsages: [AppUsage]
│   ├── appName: String
│   ├── category: String
│   └── duration: TimeInterval
└── pickupCount: Int
```

### 3.3 新增：SpendingRecord（消费记录）

```
SpendingRecord
├── id: UUID
├── amount: Decimal
├── category: SpendingCategory
├── note: String?
├── createdAt: Date
└── location: CLLocationCoordinate2D?
```

### 3.4 扩展：InferredEvent.Kind

在现有的 event kinds 基础上新增：

```
已有: sleep, workout, commute, activeWalk, quietTime, userAnnotated, mood
新增: shutter, screenTime, spending
```

所有数据源最终汇入同一条时间线，UI 层统一消费。

## 4. 快门 (Shutter) 设计

### 4.1 入口

- **常驻浮动按钮** — 首页和时间线页面右下角
- **单击** → 打开快门面板
- **长按** → 直接进入语音录制（松手结束，1 步完成）

### 4.2 快门面板

底部弹出面板，四个选项：

| 类型 | 操作流程 | 步数 |
|------|---------|------|
| 文字 | 点击 → 键盘弹出 → 输入 → 发送 | 2 步 |
| 语音 | 长按浮动按钮 → 说话 → 松手 | 1 步 |
| 拍照 | 点击 → 拍摄 → 自动保存（可选加一句话） | 2 步 |
| 视频 | 点击 → 录制（限 15 秒） → 自动保存 | 2 步 |

### 4.3 设计原则

- 记录完自动关闭面板，回到之前的页面
- 记录成功后时间线上立即出现快门事件
- 不需要选分类、打标签 — 纯粹捕捉当下

## 5. Echo 回响系统

### 5.1 内容回响（碎片重现）

将用户的快门记录在合适的时机推送提醒。

**触发策略（用户可配置）：**

| 频率 | 提醒时间点 |
|------|-----------|
| 高 | 1天 / 3天 / 7天 / 30天 |
| 中（默认） | 3天 / 7天 / 30天 |
| 低 | 7天 / 30天 |
| 关闭 | 不提醒 |

**用户设置项：**
- 回响频率：高 / 中 / 低 / 关闭
- 回响时段：用户选择接收提醒的时间段
- 单条记录可单独设置提醒时间或关闭回响

**技术实现：** iOS 本地通知（`UNUserNotificationCenter`），MVP 阶段不需要后端。

**提醒体验：**
- 推送通知：「3 天前你说："突然觉得这个项目可以..."」
- 点击进入 Echo 页面，展示完整记录 + 当时上下文（位置、天气、时间线片段）

### 5.2 关怀推送

基于数据的简单规则推送：

- 连续 3 天运动 → 鼓励
- 屏幕时间突然飙高 → 温和提醒
- 长时间没有快门记录 → 轻声问候

关怀推送可独立开关。

### 5.3 Echo 页面

App 内独立 Tab，展示：
- 今日待查看的回响
- 关怀卡片
- 历史回响记录

## 6. 仪表盘 (Dashboard) 设计

### 6.1 布局

- **顶部**：日期
- **卡片网格**：6 个核心维度的当日汇总（运动、睡眠、屏幕时间、消费、步数、快门数）
- **洞察区**：趋势对比（本周 vs 上周，今天 vs 平均值）
- **时间线预览**：最近几条事件，点击进入完整时间线

### 6.2 可定制性

- 卡片后续可由用户选择展示哪些维度、调整顺序
- 用户可添加新数据源的仪表盘卡片

## 7. 数据采集策略

### 7.1 屏幕时间

- **MVP**: 嵌入系统 `DeviceActivityReport` 视图 + 用户手动补充
- **原因**: Apple Screen Time API 限制严格，无法获取原始数据
- **注意**: 需要 Family Controls entitlement 和独立的 App Extension target，如工程量过大可推迟到 MVP 之后
- **后续**: 桌面端可更自由获取电脑端应用使用数据

### 7.2 消费记录

- 无系统 API，完全依赖用户手动输入
- 快门面板可增加「消费」快捷入口
- 输入极简：金额 + 可选分类 + 可选备注
- 后续可考虑账单 CSV 导入

### 7.3 已有数据源

- HealthKit（运动、心率、睡眠、步数、活动环）— 保持不变
- 位置服务（到访地点）— 保持不变
- 照片（PhotoKit 元数据）— 保持不变

## 9. 现有代码处理

### 复用

- HealthKit 数据采集（`HealthKitTimelineDataProvider`、`DayDataAggregator`）
- 事件推理引擎（`EventInferring` protocol 体系）
- SwiftData 模型基础（`DayTimelineEntity`、`MoodRecordEntity`）
- Protocol 抽象体系（`TimelineDataProviding`、`MoodRecordStoring`）
- 依赖注入容器（`AppContainer`）
- 位置、照片、天气数据采集

### 搁置

- Watch 端全部代码（`ToDayWatch/`）— 不删除，不维护
- 心率河流图、周报等旧版可视化组件 — 如仍存在则搁置，后续可作为可选组件回归

### 重写

- `OnboardingView` — 权限请求内容变化
- `TodayScreen` → 仪表盘
